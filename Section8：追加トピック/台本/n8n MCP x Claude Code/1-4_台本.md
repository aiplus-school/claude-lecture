## [P1] カバー — n8n MCP　自然言語でワークフローを自動生成しよう

今日は、n8n MCP を使って、日本語で話しかけるだけで業務自動化のワークフローを組み立てる方法を学びます。

n8n（エヌエイトエヌ）は、「〇〇が起きたら → △△する」を、画面の上でブロックをつないで作る自動化ツールです。たとえば「問い合わせフォームに入力が来たら、表に記録して、チャットに通知する」といった流れを、コードを書かずに作れます。しかもその流れが、**そのまま図として画面に残る**——ここが後半の話につながる大事なポイントなので、頭の片隅に置いておいてください。

とても便利なのですが、いざ自分で作ろうとすると、どのブロックを選んで、どこに何を設定すればいいのか——最初はここでつまずきます。今日は、その**組み立てと、後からの手直しを、まるごと Claude に任せる**方法をお見せします。

---

## [P2] n8n MCP とは — AIをあなたのn8nに直接つなぐ入り口

MCP、「Model Context Protocol（モデル・コンテキスト・プロトコル）」は、AIが外部のツールと連携するための共通規格です。

なぜこれが必要なのか。AIに、ただ「n8nのワークフローを作って」と頼むと、実は**それっぽいウソを書いてきます。** n8nのブロック——専門的には「ノード」と呼びます——は2,000種類以上あって、AIがその名前や設定項目を全部正確に覚えているはずがないからです。結果、存在しないブロック名や間違った設定を"それっぽく"でっち上げて、動かないものが出来上がります。

そこで使うのが、n8nに用意されている、**AIを自分のn8nに直接つなぐための入り口**——今日の主役、MCP機能です。これを有効にすると、AIはあなたのn8nの中で、ワークフローを**作り、検証し、さらにテスト実行まで**できるようになります。うろ覚えで書いた設計図を渡してくるのではなく、**あなたのn8nの中で実際に組んで、動くかどうかまで確かめてから**「できました」と言ってくる。ここが決定的な違いです。

## [P3] なぜ Claude Code とつなぐのか — 自分で作るときの"詰まり"が消える

こんな悩み、ありませんか？

n8nの画面で自分でワークフローを作ってみると、動かした瞬間にエラーが出る。**でも、その原因がまったく分からない。** エラーの画面をスクショで撮ってAIに送ったり、エラー文をコピーして貼って聞いたりしても、なかなか原因にたどり着けず、気づけばそこで何十分も溶けている……。

「じゃあ最初からAIに相談して作ろう」と考えますよね。ところが今度は、AIが**存在しないブロックを勧めてきたり、ブロックの名前が本物と微妙に違っていたり**して、言われた通りに作っても動かない。さっきお話しした「AIがでっち上げる」問題に、ここでぶつかるわけです。

この2つの"詰まり"、心当たりがある方も多いと思います。実はこれ、Claude Code と n8n を MCP でつなぐと、まとめて解消されます。理由は2つ。**ひとつは、AIがあなたの本物のn8nを見ながら話すので、存在しないブロックの取り違えが起きにくいこと。** もうひとつは、**相談した内容が、そのままあなたのn8nに反映されること。** 「AIの言う通りに自分で画面を組み直す」「出たエラーをコピーして貼って聞く」——あの往復が、そもそも要らなくなるんです。

## 【実演パート】n8n MCPでワークフローを作らせる

（スライドを閉じて、Claude Code CLIのターミナル画面に切り替え）

### まず、つなぐ

接続はとても簡単で、やることは2つだけです。

1つ目。n8nの設定画面で、**Settings → Instance-level MCP を開いて、画面右上のスイッチをオン**にする。これで、あなたのn8nがAIからの接続を受け付けるようになります。オンにしたら、隣の「Connection details（接続情報）」に**サーバーURL**が出るので、これをコピーしておきます。

（n8nの設定画面を見せて、右上のトグルをオン → Connection details でURLをコピー。※中央の「Enable MCP access」ボタンは反応しないことがある——収録は右上トグルで統一）

2つ目。Claude Code に戻って、**「/mcp」と打ちます**。コネクタを探す画面が開くので、**「n8n」と検索して「連携させる」**。さっきコピーしたサーバーURLを貼り付けて「続ける」を押すと、n8nの画面に切り替わって「Claudeにアクセスを許可しますか？」と聞かれるので、**「Allow（許可）」**を押します。

**これだけです。APIキーの発行も、設定ファイルの編集も要りません。** この手順は概要欄にも置いておきます。

つながったか、確認してみましょう。

> n8nのMCPで使えるツールを一覧してください。

（出力を確認しながら）

ワークフローを作る・動かす・検証する、といったツールが見えています。これで、AIがあなたのn8nを直接操作できる状態になりました。

### まず設計から相談する（いきなり作らせない）

いきなり「作って」と言わず、まず設計を相談します。**なぜこうするかというと、AIの提案が自分の欲しいものとズレていないかを、作る前に確かめられる**からです。

> n8nで業務自動化のワークフローを作りたいです。
> Webの問い合わせフォームに入力が来たら、その内容をGoogleスプレッドシートに1行追加して、Slackの #n8n-test チャンネルに「新しい問い合わせがありました」と通知するワークフローを作りたい。
>
> 記録用のスプレッドシートは、私のGoogleドライブにある「AI+テスト」という既存フォルダの中に、このワークフロー構築と一緒に新しく作ってください（手作業で用意するのは省きたい）。
>
> まだ作らないでください。まず、
> 1. どのノードを使うか（スプレッドシートの新規作成もどのノードで行うか含めて）
> 2. それぞれのノードで何を設定する必要があるか
> 3. 事前に私が手作業で用意しておくべきもの（認証など）
> を教えてください。

（AIの回答を確認しながら）

フォームの受け口、条件分岐、スプレッドシート、Slack——実際にあるブロックを挙げて、それぞれ何を設定すべきか説明してくれました。

ここで面白いのが、**記録用のスプレッドシートまで、AIがこの流れの中で新しく作ってくれる**ところです。しかも「AI+テスト」フォルダに入れてほしい、と伝えただけなのですが——実はGoogleスプレッドシートを作る機能には「どのフォルダに入れるか」の設定が無いんです。ところがAIは実物の仕様を確認して、それに気づき、**フォルダへ移動する処理を自分で1つ足して**帳尻を合わせてくれます。これも「実物を見て組んでいる」からできることです。

そしてもう一つ注目してほしいのが、**「Google と Slack のログイン連携だけは、あなたが n8n の画面で先に登録しておいてください」という点**です。ここは大事な注意点で、**サービスへのログイン連携だけはAIに代われません。** みなさんが自分で試すときも、ここだけは手作業になります。今日は事前に済ませてあります。

### ワークフローを生成させる

設計に納得できたので、作らせます。作ったら**その場で検証まで**させるのがコツです。

> その構成でワークフローを作ってください。
> - 記録用スプレッドシートは「AI+テスト」フォルダ内に新規作成
> - 1行目に見出し「日時 / 名前 / メール / 問い合わせ内容」を入れ、実務でそのまま使えるよう見やすく整えてください
>   （見出し行は分かりやすい日本語で。可能なら見出し行の固定や太字など、体裁も整える）
> - フォームの入力項目は「名前」「メールアドレス」「問い合わせ内容」の3つ
> - スプレッドシートには「日時 / 名前 / メール / 問い合わせ内容」を記録
> - Slack通知には送信者の名前と問い合わせ内容を含める
> 作成したら、問題がないか検証してから報告してください。
> 本番ではなくテスト用として、オフ（非アクティブ）の状態で作ってください。

（生成と検証が走る様子を見せる）

ここで一つ大事な注意です。**いきなり本番で動かさない。** いま指示に入れたとおり、必ずテスト用として、**オフ（止まった状態）**で作らせます。いきなり動く状態で作ると、意図しないタイミングで通知が飛んだりするからです。

（n8nの画面に切り替え、ワークフローが出来ているのを見せる）

n8nの画面を見ると、さっき会話しただけのワークフローが、**もうあなたのn8nの中に**出来上がっています。MCPはあなたのn8nに直接つながっているので、「設計図をコピーして貼り付ける」といった作業はそもそも発生しません。

そして「AI+テスト」フォルダを開くと、記録用のスプレッドシートも新しく出来ています。**見出しもちゃんと入っていて、そのまま使える状態**です。空のシートを自分で用意して、見出しを打ち込んで……という下ごしらえも、まるごとお任せできました。

### テスト実行までAIにやらせる

ここが今日の見せ場です。作って終わりではなく、**その場でテスト実行までAIに任せられます。**

> このワークフローをテスト実行して、結果を教えてください。テスト用のデータは適当に作ってください。

（実行結果を確認。スプレッドシートとSlackを見せる）

**テストまでしてくれました。** AIが自分でテスト用のデータを作って流してくれて——スプレッドシートに1行増えて、Slackにも通知が届いています。作るだけでなく「ちゃんと動くかどうか」まで、会話の中で確認できました。手で同じものをブロックから組んで、テストデータを用意して、流して確認して……とやると、慣れていても10〜15分はかかる作業です。

### 複雑なワークフローだと一発でいかないことも——不具合を直してみる

さて、今回は比較的シンプルなワークフローだったので、一回のお願いで、エラーもなくきれいに完成しました。ただ、正直にお伝えしておきたいことがあります。**ワークフローが複雑になってくると、毎回これが一発で完璧に決まるとは限りません。**

そこで今日は、あらかじめ**うまく動かないワークフローを一つ用意しておきました。** フォーム営業に返ってきた顧客からの返信メールを、AIが読んで、要約や次アクション・確度に加えて、**返信の熱量から「優先度」まで自動で判定**し、スプレッドシートに記録してSlackに通知する——という、さっきより複雑なものです。優先度が付くので、シート上で並べ替えれば、熱いお客さんから対応できます。これをAIに直してもらいましょう。むしろ「うまくいかないものを直す」場面こそ、n8nとAIを直接つないでいる強みが出るところです。

このワークフローは、**概要欄にJSONファイルを置いておくので、みなさんも同じものを試せます。** 手順は2つだけ。まず n8n の画面右上のメニューから **「Import from File」でそのJSONを読み込む**。これでワークフローが自分のn8nに入ります。

（JSONをインポートする様子を見せる）

もう一つ、大事な一手間があります。**AIにこのワークフローを触ってもらうには、n8n側で「AIに公開」する設定が要ります。** 設定（Settings）の **「Instance-level MCP」→「Workflows」タブ**を開いて、**「Enable workflows」ボタン**から、今インポートしたワークフローを有効にします。これをやっておかないと、AIに頼んでも「そんなワークフローは見つかりません」となってしまいます。

（設定画面で Enable workflows を押し、対象ワークフローを有効化する様子を見せる）

準備ができたので、実際に見ていきましょう。

（用意しておいた、不具合のあるワークフローを開いて見せる）

このワークフロー、動かすとエラーは出ないんです。ちゃんと最後まで流れる。でも結果を見ると、**スプレッドシートの「次アクション」の列だけ、いつも空っぽ**。他の列は埋まっているのに、ここだけ入らない。こういう「動くのに、どこか一部だけおかしい」のが、実は一番やっかいで、パッと見では原因が分かりません。さっき[P3]でお話しした「自分だけで悩む」状況、まさにこれですね。これをAIに投げてみます。

> n8nに「営業メール返信→ステータス記録（デモ用）」というワークフローがあります。
> 実行すると、スプレッドシートの「次アクション」列だけ毎回空になります。原因を調べて、修正してください。また、修正前と修正後で何が変わったのかを、
> n8nを知らない人にも分かるように説明してください。

（原因説明と修正が返ってくる）

AIはあなたのn8nの中のそのワークフローを直接読んで、原因を突き止めてくれました。データの受け渡しで、参照する場所がほんの少しズレていた——それだけのことなのですが、複雑な流れの中だと自分ではなかなか見つけられません。直してもう一度動かすと、今度は「次アクション」の列もちゃんと埋まっています。しかも、専門用語を知らなくても分かるように、何をどう直したのかまで説明してくれました。**エラーの画面をスクショで送って、コピペして原因を探して……という、さっきの往復はどこにもありません。** ここが、n8nとAIを直接つないでおくことの、いちばんの価値かもしれません。

だから、この講座で一番覚えて帰ってほしい使い方はこれです。**「土台をAIに高速で作ってもらって、複雑なところや詰まったところは会話で直していく」。** 最初から完璧を狙わなくて大丈夫です。

## [P4] クロージング — Claude Code と n8n、それぞれの良さを活かす

n8n MCP でClaudeとn8nをつなぐことで、ワークフローを**組み、検証し、テスト実行し、うまくいかないところは会話で直す**——ここまでが一気通貫で、あなたのn8nの中で完結しました。

ここで、少し引いた話をさせてください。**「そもそも Claude Code や Codex に直接プログラムを書かせれば、同じ自動化はできるのでは？」** と思った方もいると思います。そのとおりで、多くの場合できます。どちらが上ということではありません。ただ、それでも n8n がこれだけ使われているのには理由があって、大きく3つあります。

**1つ目。業務の流れが、そのまま図として見える。** どこから処理が始まり、どんな順番で進んで、どこで枝分かれして、最後にどこへ保存・通知するのか——これが一目で分かります。コードで書くと、この全体像は書いた本人にしか見えません。**エンジニアでない同僚やお客さんにも、画面を見せてそのまま説明できる。** これは実務でかなり効きます。

**2つ目。AIに任せる部分と、決まったとおりに動かす部分を分けられる。** さっきの修正デモがまさにそうでした。メールの中身を読んで分類するのはAI、金額やステータスによる分岐は決まったルール、という具合です。**ぜんぶAIに丸投げせず、必要なところにだけAIを使える。** だから結果がブレにくいですし、「どこでAIを使っているか」が図の上で見えるので、任せている範囲を人に説明しやすい。

**3つ目。複数のAIやサービスの役割分担が、目で見て確認できる。** 「ざっくり仕分けは安くて速いAI、大事な分析は性能の高いAI、保存はスプレッドシート、通知はSlack」といった使い分けができます。**使い分けること自体は Claude Code や Codex でもできます。** n8nが強いのは、**どの工程で何を使っているかが図の上で分かる**こと。コストと性能を整理しながら設計でき、その構成をチームにもそのまま共有できます。

そして今日お見せしたように、その n8n を **Claude Code が代わりに組み立ててくれる。** 「ブロックの名前や設定を調べる」というしんどい部分をClaude Codeが引き受けて、「流れが目で見えて、人に説明できる」というn8nの良さはそのまま残る。**どちらか一方を選ぶのではなく、お互いの得意なところを組み合わせる**——これが、今日いちばんお伝えしたかった形です。

### 最後に — 自分で試すときの注意点

自分で試すときの注意点があるので説明します。**最初は必ずオフ（止まった状態）で作る**こと。いきなり動く状態にすると、意図しないタイミングで通知が飛んだりします。**Google や Slack などのログイン連携だけは、自分でn8nの画面から登録する**こと。ここはAIに代われない唯一の手作業です。そして、**一発で完璧を狙わなくていい**こと。今日の1つ目のように一度で決まることも多いですが、複雑になると思ったとおりにならない場面も出てきます。そのときは今日お見せしたとおり、**土台をAIに作らせて、気になるところは会話で直していく。** これが結局いちばん早いです。

まずは、みなさんの身近な「これ毎回手作業で面倒だな」という作業を一つ、AIに相談してみてはいかかでしょうか。
