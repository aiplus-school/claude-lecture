# 1-1. 概要・実用事例のリサーチ

## 1. n8n とは何か

### 一言でいうと
**「〇〇が起きたら → △△する」を、画面上でブロックをつないで作る業務自動化ツール。**

読み方は「エヌエイトエヌ」。"nodemation"（node + automation）の略で、n と n の間に8文字あることが由来。

### 基本構造（3つの概念だけ押さえれば動く）
| 概念 | 意味 | 例 |
|---|---|---|
| **ノード（Node）** | 処理の1ブロック | 「Gmailを読む」「Slackに送る」 |
| **ワークフロー（Workflow）** | ノードを線でつないだ処理の流れ全体 | フォーム受信→記録→通知 |
| **トリガー（Trigger）** | ワークフローの起点。何が起きたら動くか | Webhook受信、毎朝9時、メール着信 |

### 立ち位置（競合との比較）
講座では「なぜ Zapier や Make ではなく n8n なのか」を必ず聞かれる。

| ツール | 連携アプリ数 | 強み | 弱み | 向く人 |
|---|---|---|---|---|
| **Zapier** | 7,000以上 | 圧倒的なアプリ数、最も簡単。まず動かしたい人の第一選択 | 実行回数課金でコストが膨らむ。細かい制御が不得手 | 初導入で最短成果 |
| **Make** | 中間 | 複雑な処理をGUIで表現できる。可視化しながら設計 | 画面が複雑で初心者に難しい。日本語情報が少ない | コストと複雑さの両立 |
| **n8n** | 500以上 | **セルフホストで無料・実行無制限**、JavaScriptで自由に拡張、オンプレ運用可 | 環境構築・認証設定に知識が要る。学習曲線が急 | 機密データ運用、長期コスト最適化、**AI連携** |

> **講座での説明のしどころ**：n8n が選ばれる理由は「無料」と「自分のサーバーに置ける」。そして **「オープンソースだからAIが仕様を読める」**＝MCPが成立する土台がある、という点。Zapier はそれができない。

### 料金（受講者が必ず気にする）
| プラン | 価格 | 実行回数 |
|---|---|---|
| **Community Edition（セルフホスト）** | **完全無料** | **無制限**。ワークフロー数・ユーザー数も無制限 |
| Cloud Starter | €24/月 | 2,500回/月 |
| Cloud Pro | €60/月 | 10,000回/月 |
| Cloud Business | €800/月 | 40,000回/月 |

- セルフホストの実コスト：月$5〜7のVPS。Cloud Starter比で年間$150〜540の節約。
- 無料セルフホスト版に500以上の連携と主要機能がすべて含まれる。有料なのは SSO・高度なRBAC・専任サポート・ログストリーミングといった企業向け機能のみ。
- Cloud の StarterとProは14日間の無料トライアルあり（クレカ不要）。

> ⚠️ **収録前に必ず確認すべき落とし穴**
> **n8n クラウド版の無料トライアル期間ではAPIキーが取得できない**（フラグメンツ記事）。APIキーがないと n8n MCP の「n8n本体に直接反映する」機能が使えない。**有料課金かセルフホストが必須。** 受講者に「無料で全部できます」と言うと詰む。

## 2. n8n MCP とは何か

### 一言でいうと
**n8n のノード仕様書をまるごとAIに読ませるための橋渡し役。**

### なぜ必要なのか（ここが講座の核心）
AIに「n8nのワークフローのJSONを書いて」と頼むと、**ノード名や設定項目を"それっぽく捏造"して動かない。** n8n のノードは2,000種類以上あり、AIが正確に記憶しているはずがない。

n8n MCP を繋ぐと、AIは**実在するノードの仕様と、既存のテンプレートを参照しながら**組み立てる。
公式の表現を借りれば「**推測ではなく事実で**ワークフローを構築する」。

### スペック（数字で語れると説得力が出る）
| 項目 | 数値 |
|---|---|
| 対応ノード数 | **約2,150**（コア826 + コミュニティ1,324） |
| ノードプロパティのカバレッジ | 99% |
| ノード操作のカバレッジ | 63.6% |
| ドキュメントのカバレッジ | 87% |
| ワークフローテンプレート | **2,352件** |
| 提供ツール数 | **23個**（コア7 + n8n管理16） |

### 提供ツール（何ができるか）
**コアツール7個（APIキー不要で使える）**
- `tools_documentation` — MCP仕様の確認
- `search_nodes` — ノードの全文検索
- `get_node` — ノード情報の取得（複数モード対応）
- `validate_node` — ノード設定の検証（最小/完全モード）
- **`validate_workflow`** — ワークフロー全体の検証 ← **MCPならではの目玉**
- `search_templates` — テンプレート検索（キーワード/ノード/タスク別）
- `get_template` — ワークフローJSONの取得

**n8n管理ツール16個（APIキー必須）**
ワークフロー管理、実行管理、データテーブル、認証情報管理、監査機能

> ⚠️ **APIキーの有無で使えるツール数が変わる。** `N8N_API_KEY` が未設定だと **38個→22個に制限**され、システム間の自動連携が使えなくなる（Qiita/nogataka）。
> ※ツール数の表記は情報源により23/30+/38と揺れがある（バージョン差と思われる）。**収録前に実物で要確認。**

### n8n-skills（MCPとは別物・見落としやすい）
MCPと**別に**、Claude Code 用のスキルが提供されている。
```
/plugin install czlonkowski/n8n-skills
```
7つのスキル（Expression Syntax、Workflow Patterns など）がクエリに応じて自動発動する。

**役割分担が明確**：
- **n8n-skills** → ドメイン知識（書き方の作法）を担当
- **n8n-mcp** → 実行手段（検索・検証・反映）を担当

> 講座で「MCPとスキルの違い」を説明できると、受講者のClaude Code理解が一段深まる。

### セットアップの選択肢
| 方法 | 手順 | 講座向き度 |
|---|---|---|
| **ホスト版**（dashboard.n8n-mcp.com） | URLを繋ぐだけ。無料枠100ツール呼び出し/日 | ◎ **最速。講座はこれ** |
| npx | `npx n8n-mcp` → 設定ファイルにURL/APIキー記入 | ○ |
| Docker / Railway / ローカル | 自由度は高いが手順が増える | △ |

**npx での手順（実測ベース／Qiita・フラグメンツ）**
1. n8n管理画面 → Settings → n8n API でAPIキーを生成
2. `.mcp.json` に `N8N_API_URL` と `N8N_API_KEY` を設定
3. Claude Code を再起動
4. （任意）`/plugin install czlonkowski/n8n-skills` でスキル追加

## 5. 講座で扱う際の注意点

- 🚨 **本番のワークフローを直接編集させない。** 必ずコピーを作り、開発環境でテストする。

## 6. 💰 受講者の費用概算（この動画の内容を自分でやる場合）

> **結論：課金は必須ではない。セルフホストなら0円で全部できる。
> 手軽さを金で買うなら Cloud Starter で月€20〜24（約3,400〜4,100円）。**

### ⚠️ 重要な訂正：APIキーがなくてもMCPは使える
以前「APIキーがないとMCPが使えない＝課金必須」と整理していたが、**これは誤り。**
公式リポジトリで確認した結果は以下のとおり。

| | APIキー**なし** | APIキー**あり** |
|---|---|---|
| **使えるツール** | **コア7個**（`search_nodes` / `get_node` / `validate_node` / **`validate_workflow`** / `search_templates` / `get_template` / `tools_documentation`） | 上記 + **管理16個**（ワークフロー作成・更新・実行・認証情報・監査） |
| **できること** | **ワークフローの設計・構築・検証（design, build, validate）** | 左記 + **n8nへの自動デプロイ** |
| **n8nへの入れ方** | 生成されたJSONを**n8nのUIに手動で貼り付け**（インポート） | `n8n_create_workflow` で**自動反映** |

> **APIキーの有無で変わるのは「最後の一手間（自動反映か、コピペか）」だけ。**
> **MCPの本質的な価値である「実在のノード仕様を参照して組み立てる」「`validate_workflow` で検証する」は、APIキーなしでも全部使える。**

**さらに、セルフホストなら自分でAPIキーを発行できる（無料）。**
→ **APIキーが欲しいこと自体は、課金の理由にならない。**

### 「無料トライアルではAPIキーが取れない」の正しい意味
n8n Cloud の14日間無料トライアルではAPIキーが取得できない（フラグメンツ）。
これが意味するのは **「Cloud のトライアルで、自動デプロイまで試すことはできない」**というだけ。
**MCP自体は動くし、JSONを手でインポートすれば動画と同じ結果にはたどり着ける。**

### 講座としての判断ポイント
- **収録はどちらでやるか？** セルフホスト（0円・受講者が真似しやすい）か Cloud（画面がきれい・手順が短い）か。

# 1-2. 「実際に使ってみる」検証アイデア 3案（提案）

> 選定基準：①受講者が価値を実感できる ②収録尺に収まる ③「n8n MCPでないと辛い」が伝わる

## 案A：問い合わせフォーム → スプレッドシート登録 → Slack通知【王道・鉄板】
- **内容**：「フォームに入力が来たら、Googleスプレッドシートに記録して、Slackに通知して」と Claude Code に日本語で頼むだけでワークフローを生成。
- **狙い**：最も一般的な業務自動化。受講者が自分の仕事に置き換えやすい。実用事例のマーケ欄と完全に一致。
- **MCPの価値が出る点**：3サービスのノード設定（項目のマッピング等）をAIが仕様通りに埋める。
- **参考数値**：類似のWebhook→IF→Slackが**やり取り5〜6回で完成、GUIなら10〜15分**（Qiita実測）。
- **想定リスク**：**Slack・Googleの認証設定はAIに代行できず、手作業で時間がかかる。** → 事前に接続を済ませ、収録では「認証は済ませてあります、ここだけは手作業です」と正直に説明する（むしろ誠実さが伝わる）。

## 案B：問い合わせ内容をAIが自動分類 → 振り分け【AI連携の魅力】
- **内容**：受信した問い合わせを OpenAI ノードで「クレーム/質問/営業」に自動分類し、種類ごとに違うチャンネルへ流す。
- **狙い**：「AIを組み込んだ自動化」という一段上の絵。n8nの2026年の目玉がAI連携である点とも合う。
- **MCPの価値が出る点**：分岐（IF/Switch）ノードとAIノードの組み合わせは手で組むと面倒。
- **想定リスク**：OpenAIのAPIキーが別途必要（**実費**）。「なぜ主ツール以外を使うのか」の説明が要る。**Gemini APIはHTTP Request Toolで既知の不具合あり**（Qiita/toki_mwc）→ 使うならOpenAIが無難。

## 案C：既存ワークフローの「検証・修正」をAIにやらせる【MCPならでは】
- **内容**：わざと壊した／不完全なワークフローを用意し、`validate_workflow` で問題点を検出させて Claude Code に直させる。
- **狙い**：**案A・Bと違い、これはMCPがないと絶対にできない。** 「AIにJSONを書かせるのとは違う」という本質が伝わる。
- **MCPの価値が出る点**：まさにここ。実在の仕様を参照した自己検証。
- **仕込みアイデア**：`{{$json.body.message}}` の **body配下ハマり**（Qiita実測の実際のハマりどころ）を仕込むと、リアルで教育効果が高い。
- **想定リスク**：地味で絵的に映えない。単体では1本にならない → **案Aの後半パートとして組み込む**。

## 推奨の組み立て
**案A（土台をAIに作らせる）→ 案C（AIに検証・修正させる）**を軸に、余裕があれば案BをAI連携の発展形として紹介。

**理由**：
1. 案Aで「速い」を体感させる（10〜15分→5〜6往復）
2. 案Cで「**なぜMCPなのか**」を納得させる（検証はMCPにしかできない）
3. 「一発では完璧にならない、改善前提」という講座方針とも噛み合う
4. 案Bは実費と既知の不具合があり、初回としてはリスクが高い
