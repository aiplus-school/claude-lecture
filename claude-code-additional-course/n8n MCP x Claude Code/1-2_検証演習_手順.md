## 1. 事前準備（収録前に済ませておく）

案A・案Cとも、以下が揃っていないと途中で止まる。**認証まわりはAIに代行できない**ので手作業で先に潰す。

### 1-1. n8n 環境
- [ ] n8n を用意（**セルフホスト＝0円**、または **Cloud Starter**）
  - ※ Cloud の**14日間無料トライアルではAPIキーが取れない** → 案Aの「自動反映」を見せるなら有料 or セルフホスト
  - ※ APIキーなしでも「JSONを手動でインポート」すれば案A自体は成立する（[1-1](1-1_リサーチメモ.md) 参照）
- [ ] n8n APIキーを発行（Settings → n8n API）※自動反映を使う場合

### 1-2. Claude Code 側
- [ ] n8n MCP を接続（ホスト版 `dashboard.n8n-mcp.com` が最速）
- [ ] `.mcp.json` に `N8N_API_URL` / `N8N_API_KEY` を設定（自動反映を使う場合）
- [ ] 接続確認：**P0** を実行し、**ツールが何個見えるか確認**
  - コア7個だけ → APIキー未設定 ／ 23個前後 → APIキー有効
- [ ] （任意）`/plugin install czlonkowski/n8n-skills`

### 1-3. 連携先サービスの認証（案Aで使う）
- [ ] **Google（スプレッドシート）** を n8n の Credentials に登録
- [ ] **Slack** を n8n の Credentials に登録し、投稿先チャンネル（例：`#n8n-test`）を用意
- [ ] 記録先の Google スプレッドシートを1枚作成（ヘッダ行：`日時 / 名前 / メール / 問い合わせ内容`）

### 1-4. 安全確認
- [ ] **本番のワークフローと隔離**された環境で行う（公式警告：本番を直接編集しない）
- [ ] テスト用のプロジェクト or ワークスペースを使う

> 💡 **収録の考え方**：認証設定は絵として地味で尺を食う。**「認証は済ませてあります。ここだけは手作業です」と正直に説明**して飛ばすのが、講座方針の「手順書にしない」に沿う。

## 2. 案A：フォーム → スプレッドシート → Slack通知 を AIに作らせる

### 全体像
```
[Webフォーム送信]
      ↓ (Webhook)
[入力が空でないか判定] (IF)
      ↓ OK
[Googleスプレッドシートに1行追加]
      ↓
[Slackの#n8n-testに通知]
```

### Step A-1. まず設計から始めさせる（いきなり作らせない）→ **P1**
**観察ポイント**：AIが `search_nodes` などを実際に叩いて、実在のノード（Webhook / IF / Google Sheets / Slack）を挙げてくるか。**認証は手作業が必要**と自分から言うか。
※ 推測っぽい回答なら **PA** で仕様参照を促す。

### Step A-2. ワークフローを生成させる → **P2**
**観察ポイント**：
- 生成後に `validate_workflow` を**自分で**走らせるか
- Webフォームのデータを `{{$json.body.xxx}}`（**body配下**）で正しく参照しているか ← リサーチで拾ったハマりどころ

### Step A-3. n8nに反映する
- **APIキーあり** → **P3**（自動反映）
- **APIキーなし** → 出力されたJSONをコピー → n8n画面で「Import from Clipboard」

### Step A-4. 実行テスト
- [ ] Webhook の Test URL にサンプル送信（n8nの「Listen for Test Event」→ フォーム or curl）
- [ ] スプレッドシートに行が増えるか
- [ ] Slack に通知が来るか
- 詰まったら → **P4**（症状を貼って修正依頼）

## 3. 案C：壊れたワークフローを AIに検証・修正させる

> **案Aで作ったものが「一発では完璧にならない」ことを、意図的に見せるパート。**
> ここが「AIにJSONを書かせるのとは違う＝MCPならでは」の核心。

### Step C-1. わざと壊す（仕込み）
案Aのワークフローのコピーを作り、以下のいずれかを仕込む。**リサーチで実際に報告されたハマりどころ**を使うと教育効果が高い。

| 仕込みネタ | 具体的な壊し方 | 何を学べるか |
|---|---|---|
| **① body配下ミス**（推奨） | スプレッドシートの参照を `{{$json.name}}` に変える（正しくは `{{$json.body.name}}`） | Webhookデータの構造。実際によくあるミス |
| ② ノード設定漏れ | Slackノードの投稿先チャンネルを空にする | 必須項目の検証 |
| ③ 型の不一致 | IFノードの条件を壊す（存在しない項目を参照） | 分岐条件の検証 |

> 収録では **①を1つだけ**仕込むのが分かりやすい。欲張って複数壊すと原因の切り分けが複雑になる。

### Step C-2. AIに検証させる → **P5**
**観察ポイント**：AIが `validate_workflow` や `get_node` を使って、**実在の仕様と照らして**原因（`body`配下の参照ミス）にたどり着くか。**推測で当てずっぽうを言わないか。**

### Step C-3. 修正させて再テスト → **P6**
**観察ポイント**：修正が当たるか。**「なぜそうなるか（Webフォームのデータは body の下に入る）」を平易に説明できるか** ← 講座の「手順書にしない」に直結。
- [ ] 再実行して、今度はスプレッドシートの名前欄も埋まるか確認
