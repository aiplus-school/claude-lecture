# 2-1. 概要・実用事例のリサーチ

## 1. Remotion とは何か

### 一言でいうと
**React（Webサイト作りの技術）で動画を作るフレームワーク。**
Premiere や CapCut のようなタイムライン編集ソフトではなく、**コードで動画を書く**。

### 考え方の核
> **動画 = 時間に対する画像の関数**

Remotion は「今、何コマ目か」という数字と**白紙のキャンバス**を渡してくれる。そこに好きなものを描く。コマ番号に応じて中身を変えれば、それがアニメーションになる。

書いたReactコンポーネントを1コマずつ画像として描画し、最後に FFmpeg で結合して MP4 / WebM / HEVC / GIF に書き出す。

### 秀逸な言い換え（講座で使える）
> **「Remotionは編集ソフトじゃない。動画を"ビルド"するレンダリングエンジンだ」**（note/さかもとたくま）

動画制作を「タイムライン作業」から**「フロントエンド開発の問題」**に引きずり出した、というのがRemotionの本質。コンポーネント、state、データ、API、CSS、Canvas、SVG、WebGL、アルゴリズムで動画を制御できる。

## 2. 基本の構成要素（講座で最低限触れる部分）

| 要素 | 役割 |
|---|---|
| **Composition** | 動画の設定（幅・高さ・総コマ数 `durationInFrames`・フレームレート `fps`）とReactコンポーネントの組み合わせ。`src/Root.tsx` に登録すると動画として認識される |
| **`useCurrentFrame()`** | 「今が何コマ目か」を取得。**これがRemotionの心臓部** |
| **`useVideoConfig()`** | 幅・高さ・fps・総コマ数を取得 |
| **`AbsoluteFill`** | 画面いっぱいに要素を重ねて配置する部品。レイヤー構造を作る |
| **`Sequence`** | 特定のコマ範囲だけ表示する。動画の「シーン」を作る |
| **`Series`** | シーンを順番に並べる |
| **`OffthreadVideo`** | 既存の動画を読み込む（レンダリング最適化版） |
| **`<Player>`** | 書き出さずにReactアプリ内で動画をプレビューできる部品。HTMLの`<video>`タグに似たAPI |

**動画の4つのプロパティ**：幅（px）/ 高さ（px）/ 総コマ数（`durationInFrames`）/ フレームレート（`fps`）
※ 最初のコマは **0**、最後は **`durationInFrames - 1`**。

### ⚠️ Reactの常識が通じない罠（重要・講座の見どころ）
| 罠 | 理由 |
|---|---|
| **`useState` が使えない**（too many renderings になる） | 各コマは独立して描画される。状態は持てず、**すべてを `frame` から計算する**必要がある |
| **CSSアニメーションが使えない** | 時間の流れをCSSに任せられない。時間はRemotionが管理する |
| **プレビューと書き出しは別物** | Studioはリアルタイム再生なので重い描画は遅れて見える。`remotion render` はフレームごとに止めて撮る |

> **講座での価値**：これは「Reactを知っている人ほどハマる」点。**AIに任せるとこの罠を回避できる**（Agent Skillsが29種のベストプラクティスを知っているため）＝Claude Codeと組む理由の補強になる。

## 4. なぜ「Claude Code × Remotion」なのか（講座で語る価値）

### Remotion単体の最大の課題
**Reactが書けないと何も作れない。**
動画を作りたい人と、Reactが書ける人は、ふつう別の人。**ここが最大の参入障壁だった。**

### Claude Codeと組むと
日本語で「こんな動画を作って」と頼めば、AIがコードを書いて動画を生成する。
**Remotionの参入障壁そのものが消える。**

しかも Agent Skills が入っていれば、上記の「Reactの常識が通じない罠」（useState不可、CSSアニメ不可）もAIが回避する。

> **3テーマの中で最も物語が明快。**
> **「今まで一部のエンジニアしか使えなかった道具が、全員の道具になる」**

### ⚠️ ただし致命的な注意
**案A（1本作る）だけを見せると「他のAI動画生成ツールでよくない？」で終わる。**
Remotion の存在理由は次章。

## 5. Remotion の本当の強み ＝「量産」と「データ連動」

**1本の凝った動画を作るなら、普通の編集ソフトのほうが速い。**
Remotion が圧勝するのは**同じ型の動画を大量に作る**場合。コードなので、データを差し替えれば動画が量産できる。

| 事例 | 内容 |
|---|---|
| **Spotify「Wrapped」型** | ユーザーごとの年間データを可視化した**パーソナライズ動画**を人数分生成 |
| **カンファレンス** | 発表者紹介動画を**100本一括生成** |
| **EC** | 商品DBと連携して商品紹介動画を自動生成 |
| **SNS / レポート** | 株価・天気などのデータを自動で動画配信 |
| **個人開発** | Remotion + LLM で「ゆっくり解説動画」生成システムを構築（Qiita/sumihiro3） |

### 実導入事例：GMO（One Tech Blog）★最重要
**何を作ったか**：インタビュー動画30本以上に、スコアのオーバーレイ表示と質問字幕を一括挿入。

**実装の妙**：
- **Markdownを React コンポーネントに変換**する構成
- 字幕ファイルは「00:40」のような**秒単位のタイムコード**で書き、それをコマ番号に変換
- → **非エンジニアがfpsを理解せずに台本を書ける**
- `<AbsoluteFill>` で動画に字幕を重ね、`useCurrentFrame()` でどの字幕を出すか判定

**結果**：
- **30本を約14.75時間で、夜間に無人で完走**
- **9回のクラッシュを自動で処理し、手作業の介入はゼロ**

**大変だったこと（＝講座の注意点の一次情報）**：
| 問題 | 詳細 | 解決策 |
|---|---|---|
| **ディスク容量** | バンドル処理がディレクトリを丸ごとコピーし、**4K動画1本あたり数GB**を消費 | バンドル後にディレクトリをシンボリックリンクに置換 |
| **メモリクラッシュ** | 並列Chromiumレンダリングで**OOM Killer**にプロセスが落とされる | 並列数を下げ、自動再起動ループとスキップ機能を組み合わせ |

**良かったこと**：CSSベースのスタイリングとホットリロードのプレビューがWeb開発と同じ感覚 / スクリプトを夜間に放置できる / Markdown入力なので非技術者もコンテンツを準備できる

## 6. パフォーマンス（収録の計画に直結）

| 項目 | 実測値 | 出典 |
|---|---|---|
| ローカルレンダリング速度 | **1分の動画 ≈ 約20分** | 検索結果 |
| 並列数の効果（5分動画） | concurrency=2 → **18分** / concurrency=8 → **7分** | ターキー |

**高速化の手段**：`mediacache` サイズの指定 / `OffthreadVideo` のスレッド数調整 / `concurrency` 値の調整

> 🚨 **収録への影響**：**レンダリングは想像以上に遅い。**
> → 検証・収録では**尺を短く（15秒程度）、解像度を抑える**。長尺は待ち時間で番組が死ぬ。
> → 量産デモも**5本程度に抑え、「100本でも同じ仕組み」と口頭で補足**する。

## 7. ライセンス・料金（🚨 3テーマ中もっとも慎重な説明が必要）

**Remotionは「無料のOSS」ではない。会社で使うと有料になる。** ここを曖昧にすると受講者に迷惑がかかる。

### 無料で使える条件
**個人、および3人以下の企業は無料**（商用利用も可）。登録不要で即開始できる。

### 有料が必要になる条件
**営利企業が4人以上になった時点で、企業ライセンスが必須。**

### 有料プラン
| プラン | 対象 | 価格 |
|---|---|---|
| **Remotion for Creators** | 低ボリュームの制作者・個人開発者 | **$25 / シート / 月**。1シート＝Remotionのコードを自分で書く人、**またはエージェント型コーディングツールを使う人** |

| **Enterprise** | 高度なニーズ | **最低$500/月**。専用Slack/Discord、月1回のコンサル、カスタム条件 |

> ⚠️ **情報の食い違いあり**：Creators の最低額について、remotion.pro は「**最低$75/月（3シート）**」、別の情報源は「**単体購入なら最低シート数・最低額なし**」としており、**矛盾している。**
> → **収録前に [remotion.dev/docs/pricing](https://www.remotion.dev/docs/pricing) と [remotion.pro/license](https://www.remotion.pro/license) で必ず最新を確認すること。**

> 💡 **講座での重要ポイント**：「エージェント型コーディングツールを使う人」もシート対象。つまり**「Claude Codeに書かせるから自分はコードを書いていない」は免除理由にならない。** まさに本講座の受講者が該当する。**必ず概要欄に明記すべき。**

# 2-2. 「実際に使ってみる」検証アイデア 3案（提案）

> 選定基準：①Reactが書けなくても作れることを示す ②Remotionでないと辛い（＝量産・データ連動）が伝わる ③レンダリング時間が収録尺に収まる

## 案A：日本語の指示だけでショート動画を1本作る【つかみ】
- **内容**：`npx skills add remotion-dev/skills` を入れた状態で、Claude Code に「〇〇を紹介する15秒の縦型ショート動画を作って」と日本語で頼み、コードが書かれて動画が書き出されるまでを見せる。
- **狙い**：**「コードが書けなくても動画が作れた」**という驚きを最初に置く。受講者の心をつかむパート。
- **Claude Codeの価値が出る点**：Remotion の参入障壁（React）を丸ごと飛ばせる。しかもAgent Skillsが `useState` 不可などの罠を回避する。
- **見せ場**：AIが書いたコードをチラっと見せて「`useCurrentFrame()` でコマ数から位置を計算している」と**理由**を説明する（＝講座方針「手順書にしない」）。
- **想定リスク**：🚨 **これ単体だと「AI動画生成ツールでよくない？」で終わる。** 必ず案Bとセットで「量産」まで見せる。
- **尺の目安**：15秒動画 → レンダリング数分程度。**編集で待ち時間をカット**。

## 案B：データを差し替えて同じ型の動画を一括量産【Remotionの本命】
- **内容**：スプレッドシートやJSONの一覧（例：メンバー紹介、商品一覧、ランキング）を渡し、**同じデザインで中身違いの動画を5本まとめて生成**させる。
- **狙い**：**Remotionの存在理由そのもの。** 「1本作る」ではなく「100本作れる」が伝わる。Spotify Wrapped、発表者紹介100本の事例と直結。
- **Claude Codeの価値が出る点**：データを読み込んでループさせる部分のコードをAIが書く。手作業なら1本ずつ編集ソフトで作る地獄。
- **語れる事例**：GMOの「30本を夜間14.75時間で無人完走」。**これを話すと一気に現実味が出る。**
- **想定リスク**：本数を増やすとレンダリング時間が伸びる。**5本に抑え、「100本でも仕組みは同じ」と補足。** 並列数を上げるとメモリクラッシュのリスク（GMOでも9回発生）。
- **尺の目安**：短尺5本なら現実的。

## 案C：既存の動画にテロップ／字幕を自動で乗せる【実務直結】
- **内容**：既存の動画素材に対して、字幕やスコア表示などのオーバーレイをコードで自動挿入する（**GMOの実導入事例の再現**）。Markdownのタイムコードから字幕を生成する構成にすると事例に忠実。
- **狙い**：「ゼロから動画を作る」より**現実の業務に近い**。受講者が自分の仕事に置き換えやすい。
- **Claude Codeの価値が出る点**：`<AbsoluteFill>` と `useCurrentFrame()` で字幕を出し分ける処理をAIに書かせる。
- **想定リスク**：
  - 動画素材の用意が必要。工程が多く**尺が膨らむ**。3案で最も難易度が高い。
  - **4K動画だとバンドルで数GB消費**（GMOの実例）。素材は低解像度・短尺に。
  - `OffthreadVideo` 絡みでレンダリングがさらに遅くなる。

## 推奨の組み立て
**案A（1本作って驚かせる）→ 案B（量産して本当の価値を見せる）**の2段構成。

**理由**：
1. 案Aだけだと「他のAI動画ツールとの違い」が出ず、**講座方針の「主ツール以外を使う理由・メリットを説明する」に答えられない。**
2. 案Bまで見せて初めて「**データ×コードだから量産できる**」という他ツールにない強みが立つ。
3. GMOの実例（30本無人完走）を語れば、案Bの延長線上に実務があると示せる。
4. 案Cは尺・難易度・レンダリング負荷の三重苦。**紹介のみ or 次回送り**を推奨。
